## Overall Impression

This is principled, honest product UI with unusually good engineering discipline — one status vocabulary, color never the only signal, last-good polling, explicit "—" sentinels instead of fake zeros. The biggest opportunity is that the _default_ visual contract under-delivers on contrast and structure: text-faint body text and panel/table/card boundaries are below WCAG thresholds, and the accessible version is gated behind prefers-contrast. Make the default the accessible one.

## What's Working

1. Single source of truth for status (glyphs.ts → tokens.ts → CSS var) → structural CLI/web parity; chips always carry glyph + label so color-blind operators lose nothing. Rare discipline.
2. Honest data states: last-good-on-error polling, "stale" banner, "—" for unparseable times (never fake "0s"), no live countdowns faked from a monotonic clock.
3. A11y scaffolding above the norm: 8:1 focus-visible ring, role/aria-label on chips, aria-current nav, prefers-reduced-motion + prefers-contrast escalation, aria-invalid + role=alert on form errors, empty/loading states on every view.

## Priority Issues

**[P1] Default contrast under-delivers (text + structure).** `--text-faint` body text (#64748b: event timestamps, phase hints, the "cockpit" subtitle) is 3.3–4.0:1 — fails AA 4.5:1. Structure relies on 1.10–1.12:1 surface steps and 1.34–1.99:1 borders, all below the SC 1.4.11 3:1 non-text-contrast bar for meaningful boundaries. The accessible values only exist behind `prefers-contrast: more`, which almost no one enables.

- Why it matters: on a dense dark ops dashboard, faint timestamps and near-invisible panel/table/kanban boundaries raise scanning effort exactly when an operator is triaging incidents.
- Fix: raise `--text-faint` to ≈#8b9bb0 (≥4.5:1 on panel), lift `--border`/`--border-strong` and the bg→panel→elevated steps so structure clears ~3:1 by default; reserve `prefers-contrast: more` for a further bump.
- Suggested command: /impeccable audit then /impeccable polish

**[P2] No responsive / structural breakpoints.** Zero `@media` width queries; responsiveness rests entirely on two `auto-fit minmax()` grids. The 6-column `.data-table` (incl. workspace paths) has no `overflow-x` container, and the inline top bar (brand + 4 nav links) doesn't reflow → horizontal overflow on tablet/mobile. The product register explicitly calls for responsive tables.

- Fix: wrap tables in an `overflow-x:auto` region; collapse the bar/nav at a breakpoint; consider a stacked/card table on narrow widths.
- Suggested command: /impeccable adapt

**[P2] Dead "shared" component: StatusChip.tsx.** No view imports it — Fleet and Kanban inline their own badge markup to support the unknown/"phase=" drift case, so there are two chip implementations. The shared-primitive intent (#68) and reality have diverged.

- Fix: delete StatusChip, or extend it to cover the drift badge and have both views consume it (DRY).
- Suggested command: /impeccable extract

**[P3] Flat visual hierarchy.** Every surface is the same `.panel` (same border, shadow, title weight, elevation); the primary running-fleet table reads at the same level as additive side panels (Restore, Rate limits). With near-invisible surface steps the eye gets little guidance about what's primary.

- Fix: give the Fleet table/metrics more prominence (stronger surface, larger numerals); de-emphasize additive panels.
- Suggested command: /impeccable layout

**[P3] Thin help / copy.** No tooltips or unit/effect hints on Settings knobs ("Max retry backoff (ms)"), no first-run guidance beyond "Waiting for the first snapshot…", the Kanban count-only "Claimed" column ("N reserved, awaiting dispatch") invites "why no cards?", and raw API errors ("request failed with status 503") aren't actionable.

- Fix: add field helper text, a first-run hint when no snapshot has ever arrived, an explainer on Claimed, and humanized error messages.
- Suggested command: /impeccable clarify

## Persona Red Flags

**Alex (on-call operator, power user):** During an incident the primary actions are pause dispatch / cancel a runaway session — yet Pause lives only in Settings (the 4th route), two hops from the default Fleet view; there's no global pause on Fleet/Kanban. Cancel and Retry fire with no confirmation and no undo — one misclick irreversibly cancels a long session. After acting, the button shows "requested ✓" but real success only appears on the next poll. No keyboard shortcuts (no `p` to pause, no `g f`/`g k` view jumps). The fixed 1200px max-width wastes ultrawide screens operators favor.

**Jordan (first-time operator):** Lands on Fleet showing "Waiting for the first snapshot…" with no hint that the daemon must be running. The "unknown" badge renders `phase=<internal string>`, leaking jargon. "Claimed", "Retrying", "drifted phase", and raw event `kind` strings assume domain knowledge with no in-product explanation.

## Minor Observations

- Chips use `role="img"` with an aria-label that duplicates the visible label — the visible glyph+text already conveys it; `role="img"` may be redundant/over-announced.
- Event-feed rows use a fixed `1.2em 6em auto auto 1fr` grid; the message column compresses hard on narrow widths.
- `--shadow` is applied to every panel uniformly; with the flat hierarchy it adds cost without separating layers.
- The metric row sits visually close to the "hero-metric template" — fine today because they're small functional counters, but resist any urge to enlarge + accent them.

## Questions to Consider

- What would a Fleet view look like if pause + cancel were one keystroke away, so an operator never leaves the overview during an incident?
- Should the default theme simply BE the prefers-contrast values, with the current low-contrast as an opt-in "dim" mode rather than the reverse?
- Does an operator dashboard built for triage want a confirm step (or a 5-second undo) on Cancel, the way Gmail/Linear treat destructive actions?
